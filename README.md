# HCMC SUMO Traffic Signal Control

This repository runs a SUMO traffic-signal-control experiment from one network file. It builds traffic-light metadata, generates train/test OD scenarios, evaluates fixed-time and MaxPressure baselines, trains a Dynamic Directed Graph-MAPPO-BC controller in real SUMO, then plots and reports the final test comparison.

## Input

The only required source input is:

```bash
map/net/network.net.xml
```

No hand-written traffic-light metadata, OD matrix, route file, or result file is required. These are generated by scripts:

- `map/metadata/*.json`
- `map/scenarios/`
- `results/`

The package name is `hcmc-tsc` and the import package is `hcmc_tsc`.

## Setup

Install with `uv`:

```bash
uv pip install -e .
```

SUMO must be installed separately and available as `sumo` on `PATH`.

## Config

Default experiment settings live in:

```bash
configs/experiment.yaml
```

`scripts/build_scenarios.py` reads this file by default. The YAML controls default train/test counts, duration, scale, and the family-count templates:

- `scenario_build.train_family_counts`
- `scenario_build.test_family_counts`

CLI values such as `--train-count`, `--test-count`, `--duration`, `--seed`, and `--scale` override the YAML values. If the requested count differs from the template total, family counts are scaled proportionally and rounded deterministically.

## Build Metadata

```bash
uv run python scripts/build_metadata.py
```

This parses `map/net/network.net.xml` and writes:

- `map/metadata/network_metadata.json`
- `map/metadata/demand_edges.json`

## Build Scenarios

Default build from YAML:

```bash
uv run python scripts/build_scenarios.py
```

Explicit thesis/default build:

```bash
uv run python scripts/build_scenarios.py \
  --train-count 60 \
  --test-count 28 \
  --duration 3600 \
  --seed 42 \
  --scale 1.0
```

The scenario index is:

```bash
map/scenarios/scenario_index.csv
```

There is no duplicate JSON index.

## Demand Model

The demand generator creates SUMO-valid OD routes from the network graph. It does not use `randomTrips`, cached routes, fake metrics, or measured link-level turning counts.

The current demand model is `corridor_priors_v1`: corridor-weighted realistic priors for the HCMC Võ Thị Sáu, Điện Biên Phủ, District 1 and District 3 corridor. Public link-by-link hourly turning counts for this exact corridor are not assumed to be available, so the generator uses:

- road-tier weights for named primary, secondary, and local collector roads
- boundary source/sink priors from the SUMO network
- 15-minute time-bin profiles
- family-specific demand multipliers
- AM/PM directional commute logic
- rain, holiday, night, airport-edge, and incident modifiers
- SUMO-valid shortest routes only

Default train family template:

```text
base_midday: 14
am_peak: 14
pm_peak: 14
mild_rain: 10
friday_pm_leisure: 4
stress_overload: 4
```

Default test family template:

```text
holiday_low: 6
incident_lane_loss: 6
rainy_peak: 4
night_low: 4
saturday_mixed: 4
airport_holiday_edge: 4
```

Each `scenario.json` and CSV row includes audit fields such as `demand_model_version`, `requested_vehicles`, `routed_vehicles`, `route_rate`, `time_bin_counts`, `direction_split_summary`, and `road_tier_summary`.

## Train Proposed

```bash
uv run python scripts/train_proposed.py \
  --output results/proposed/train \
  --total-updates 60 \
  --rollout-steps 128 \
  --bc-scenarios 6 \
  --bc-epochs 8 \
  --device cpu
```

Behavior cloning uses a MaxPressure teacher on train scenarios only. PPO fine-tuning also uses train scenarios only. No test scenario is used for training or checkpoint selection.

## Resume Training

```bash
uv run python scripts/train_proposed.py \
  --output results/proposed/train \
  --total-updates 60 \
  --rollout-steps 128 \
  --bc-scenarios 6 \
  --bc-epochs 8 \
  --device cpu \
  --resume results/proposed/train/checkpoints/last.pt
```

## Evaluate Proposed

```bash
uv run python scripts/evaluate.py \
  --method proposed \
  --checkpoint results/proposed/train/checkpoints/last.pt \
  --split test \
  --output results/proposed/test \
  --overwrite
```

Use `last.pt` or `best_train.pt`. Do not choose checkpoints from test metrics.

## Methods

- `fixed`: deterministic fixed-time cycling over valid SUMO green phases for each controllable traffic light.
- `pressure`: deterministic adaptive MaxPressure with min/max green, action masks, downstream spillback penalty, switch penalty, and starvation bonus.
- `proposed`: Dynamic Directed Graph-MAPPO-BC with a shared actor, centralized critic, directed graph attention, dynamic action masks, MaxPressure behavior-cloning warm-start, and PPO fine-tuning in real SUMO.

## Observation

The controller uses action-level aggregated lane/link features. For each traffic light and candidate phase/action, the observation includes incoming queue, vehicle count, waiting time, downstream queue, downstream occupancy, pressure, served-link count, turn mix, downstream spillback indicator, current-action indicator, elapsed green, valid-action indicator, and an existence/bias feature.

This is not a raw lane-level tensor. Lane restrictions and vehicle classes are still simulated by SUMO; the controller sees their effects through queue, wait, occupancy, pressure, and spillback features.

## Splits And Leakage

There is no validation split. The final comparison is test-only:

- `fixed` uses test scenarios.
- `pressure` uses test scenarios.
- `proposed` trains only on train scenarios.
- `proposed` is evaluated on test scenarios only after training.
- Checkpoint selection is based on training reward only, never test metrics.
